# Conversational Admin Dashboard — Steel Thread

## Objective

Build a chat-based admin dashboard: an admin types natural language, an LLM (via OpenRouter) with tool-calling maps the request to Convex queries, and the results render inline as streamed text plus a real chart, with a live tool-status indicator.

The Convex backend is given and live (`https://utmost-poodle-412.convex.cloud`). We build only the frontend in `src/` and consume the generated typed `api`. Full reconnaissance and the backend domain map live in `docs/REPO_TOUR.md`.

**Phase 1 is one vertical slice** that proves every integration seam exactly once, using a single tool. **Phase 2 is horizontal breadth** on that proven spine: more tools, a mutation, more chart types, and the polish that makes it feel like a real operator console. A few decisions are still open — see Open Questions at the bottom.

<Callout id="done-definition" tone="decision">

**"Phase 1 done" means:** the full seam chain below runs end to end with one tool, the gate stays green on every commit, and the Vitest + Playwright suites pass. That covers 7 of the 14 brief criteria. The remaining 7 are deferred to Phase 2 on purpose — see the coverage tables.

</Callout>

## Working methodology (every commit)

This is a take-home, and the commit/PR history is itself a grading signal — end-loaded tests are a known anti-pattern, so coverage ships alongside the logic it covers.

- **Atomic, mise-en-place commits.** Prep the pure ingredients first, then wire them in. Each commit tells one part of the story and keeps the gate green (`lint-staged` + `bun run lint`).
- **Test-alongside, never end-loaded.** A commit that adds non-trivial logic ships its own coverage (unit or e2e, whichever fits the seam). Prep/scaffolding commits, and commits only testable once a later piece exists, may legitimately carry none.
- **File-structure convention** (per `AGENTS.md`): imports → declarations → prep data → helper/pure fns → main orchestration at the bottom. Build leaves to root, following Grokking Simplicity: data → calculations → actions.
- **Dependency injection is the enabler.** `loop.ts` and each tool's `run` take their dependencies (OpenRouter fns, Convex client) as parameters, so they test in isolation with fakes. No hard-imported singletons in the orchestration path.
- **TypeScript discipline.** No `any`, no `@ts-ignore`, no type-loosening to compile — fix the type. Annotate data-flow boundaries with `// -> Type` comments on the lines that cross a transformation.

## Build order — tiers vs commits (one PR: "steel thread")

There is no monolithic tooling commit: each dev/runtime dependency is installed just-in-time, in the commit that first needs it (Vitest before the first unit test, Recharts before the chart, Playwright before the e2e). ## Testing approach

- **Unit (Vitest):** the pure calculations — `validate`, the OpenRouter parsing. Fast, offline, deterministic.
- **Integration (Vitest):** `loop.ts` with injected fakes; assert state transitions and the error path.
- **E2E (Playwright):** mock the LLM (non-deterministic, costs tokens), real Convex (proves the typed-`api` wiring end to end, like a REST integration test against a real test DB).
- **Behavior, not implementation:** assert what the admin experiences (pill shows, text streams, chart renders), not that a function was called.
- **Edge cases, as each seam lands:** OpenRouter request fails, Convex call fails, LLM returns invalid args, LLM names an unknown tool. Each surfaces gracefully in the chat, never crashes.

<Callout id="no-tsc-gate" tone="decision">

**No whole-project `tsc` gate.** The given `convex/` source does not type-check, and the generated `api.d.ts` pulls those modules into any `tsc` run, so a clean compile is impossible without editing given code. Type safety on `src/` comes from type-aware ESLint plus the editor's TypeScript server.

</Callout>

## Phase 2 — breadth on the proven spine

Each item is a small vertical (data → calculation → action → UI → test) on the spine Phase 1 already proved. The rationale for every decision is recorded in `DESIGN.md`, written incrementally (one entry per decision as its commit lands), then a final editing pass.

### The multi-step agentic loop

The Phase 1 `while` loop generalizes to real multi-turn tool-calling. Name → id resolution is **LLM-driven** (the model calls `listAll`, reads the result, then calls `pause` with the resolved id) — our code never string-matches names; `validate` throwing on an unknown id is the safety net. Tool-layer errors are fed back to the LLM as that tool's result so it can self-correct; only an unreachable-LLM error aborts the turn. A `MAX_STEPS` cap (~5) guarantees termination and bounds per-message cost.

<Mermaid
  id="agentic-loop"
  caption="Phase 2 multi-step loop. Tool errors re-enter the LLM as results; only the step cap or a final answer exits."
  source={
    "flowchart TD\n  A[User message] --> B[Build LLM context]\n  B --> C{LLM: tool call or answer?}\n  C -->|answer| J[Stream answer + render]\n  C -->|tool call| D[validate args]\n  D -->|valid| E[run Convex fn]\n  D -->|throws| F[feed error back as tool result]\n  E --> G[feed result back as tool result]\n  F --> H{steps < MAX_STEPS?}\n  G --> H\n  H -->|yes| C\n  H -->|no| I[graceful, reason-bearing stop]"
  }
/>

<Callout id="no-chained-mutations" tone="warning">

**No chained mutations.** Flows are reads-then-one-terminal-write, so there is never a half-applied mutation to roll back. If two writes ever had to be atomic, that belongs in a single Convex mutation — not two tool calls the LLM chains.

</Callout>

### Mutation UX

- **`pause` / `resume`** fire directly; the LLM acknowledges the result ("Paused Daily Project Accounting"). Reversible, so no grace period.
- **`enqueue`** carries a 5-second undo-send window (`undoWindowMs: 5000`): the loop defers the Convex call behind a client-side timer, so nothing reaches the backend until it elapses. Undo clears the timer; both fire and cancel feed a tool result back to the LLM so it closes the turn. Delivery is inert on the preview deployment (no channel credentials per `API.md`), so it is risk-free to demo.
- **Ambiguity (Should #8)** is handled at the prompt level for all tools: when the target is unclear ("which task?"), the LLM asks rather than guesses. This is separate from the mutation grace period.

- **Tables render in-chat** via `react-markdown` + `remark-gfm`: the LLM emits GFM table syntax, rendered as a React subtree (safe inline, no raw HTML, no sandbox). **Charts** render as Recharts components keyed off the typed tool result, not through markdown.
- **Conversation context (Should #7)** keeps full prose continuity, but once a turn completes, bulky tool-result content is stubbed (e.g. `[chart data rendered]`) to bound token growth. We stub rather than delete because the API rejects a `tool_call` with no matching tool-result message. Windowing + summarization is the production scale-up, omitted here because admin sessions are short (noted in `DESIGN.md`).
- **Time-window vocabulary** mirrors the existing tool's filters (`24h / 7d / 30d / 60d / 90d`): the LLM maps "this week" → 7d, "this month" → 30d, and `days: 90` stays the widest window so seeded data is always in range.

### Design direction

The dashboard is internal admin/operator tooling, not contractor-facing ("the admin is the user"; the backend has no per-tenant scoping). Match the existing "MonsterClaw" ops-console aesthetic: near-black background, construction-orange primary (`#F96302`), monospace labels, sharp corners. Source components from the `phillips-poc-public` project (Tailwind 4, shadcn/Radix, lucide, `--radius: 0`), reusing its dark-mode CSS variables and swapping `--primary` to the orange.
